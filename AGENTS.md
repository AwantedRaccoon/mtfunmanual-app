# MTF App Agent contract

## 作用域与优先级

- 本文件适用于 `mtf-app/` 的整个目录树；更深层的 `AGENTS.md` 可以为子模块补充更具体的规则。
- 用户在当前任务中的明确要求高于本文件。规则冲突时，优先遵守更具体、离目标文件更近的规则。
- 本项目已确定先做 iOS 原生版本：SwiftUI + SwiftData，最低 iOS 17.0，同时支持 iPhone 与 iPad。首发功能仍以 `docs/product/` 中的规划为准；不要把未确认细节伪装成既定决定。
- 当项目变大时，把架构、产品和视觉细节移入 `docs/`，让根 `AGENTS.md` 保持为入口地图和不可越界合同。

## 项目定位

- 这是 MTF不全书的 App，不是网站套壳。它把网站工具箱中适合随身使用的功能改造成更私密、更容易完成的移动端工具。
- 核心体验是：温和、可信、克制、可暂停。用户不需要先证明身份，也不应被催促做医疗决定。
- 核心表达是“先不要急着给自己下结论。”它是产品判断原则，不是每个页面都要重复的装饰文案。
- 首要用户可能正在焦虑、信息不足、缺乏医疗资源或需要隐私保护。流程必须减少认知负担，明确下一步，也允许随时退出。
- App 不是身份测试、处方平台、药品商城、社交广场或泛用型健康仪表盘。未经产品决策，不要自行扩展成这些产品。

## 当前视觉来源与继承边界

- 唯一有效的品牌视觉参考是同级仓库 `../mtfbook.com` 的 `main` 分支根目录。
- 优先查看 `../mtfbook.com/styles.css`、`sections.css`、`article.css`、`hrt.css`、`hrt-calculator.css` 以及对应的当前 HTML/JS。
- V1 已退役。禁止切换、复制或参考 `archive/v1-retired-20260717`；禁止复建 `V2Preview`、`V2拼装版` 或历史实验目录。
- 继承的是颜色、字体角色、材料感、信息语气和交互原则，不是网页 DOM、桌面导航、十二栏结构或具体区块顺序。
- App 的信息架构必须从触摸、单手操作、系统返回、离线/弱网和安全区重新设计。禁止把网页缩窄后称为 App。

## 美术总纲

当前视觉可概括为“数字纸本编辑系统”：浮世绘数字风只保留克制的色彩与平面构成，主体更接近一本带有目录、票据、档案页和校样批注的当代中文刊物。

设计应同时具备四种感受：

1. **纸张感**：温暖、可停留，不像冷白色医疗后台。
2. **墨色秩序**：深靛承担主要结构、文字和可信度。
3. **编辑痕迹**：编号、细则线、页签、票据、边注帮助定位，不做无意义装饰。
4. **有限冲突**：朱红或芥末金只在判断、提醒和行动处打破安静。

每个主要屏幕只需要一个明确视觉主张。宁可留出大块纸面，也不要用一组相似圆角卡片填满空间。

## 品牌色令牌

新代码必须把颜色集中为语义令牌；不要在组件里散落未经说明的新色值。当前基准色为：

| 令牌 | 色值 | 角色 |
| --- | --- | --- |
| `rice` | `#efe6d4` | 页面主底色、安静的外层纸面 |
| `paper` | `#f7f0e3` | 内容纸张、表单、抽屉和阅读面 |
| `indigo` | `#172f4a` | 主文字、边框、实心按钮 |
| `indigoDeep` | `#0f2135` | 导航、沉浸区和最高对比背景 |
| `vermilion` | `#b94d42` | 关键判断、警示、强调和主行动 |
| `rose` | `#dca59b` | 情绪、关系与较柔软的辅助面 |
| `mustard` | `#c5a441` | 选中态、焦点、编号和可信提示 |
| `blue` | `#477594` | 医疗、身体、研究路径和信息面 |
| `moss` | `#78815b` | 长期照护、缓解和生活内容 |

- 默认正文使用 `indigo` 或 `indigoDeep`，不要用浅灰削弱可读性。
- 一个屏幕通常使用米纸/纸白、靛蓝和一个强调色；不要同时把全部品牌色当彩虹使用。
- 深色区以 `paper` 为正文色、`mustard` 为定位色；朱红只承担少量高价值信号。
- 新增类别色之前先证明现有令牌无法表达其语义，并更新此表和深浅色对比测试。

## 字体角色

- **展示字**：`Songti SC`、`STSong`、`SimSun` 或获许可的同类中文宋体。用于主标题、章节标题、关键数字和短句。
- **正文字**：`PingFang SC`、`Hiragino Sans GB`、`Microsoft YaHei` 或平台系统无衬线。用于正文、表单和操作说明。
- **功能字**：`Arial Narrow`、`Helvetica Neue`、`Arial` 或同类窄体无衬线。用于英文眉题、编号、日期、状态和微型标签。
- 原生端应优先使用平台字体和动态字号；若引入字体文件，先确认中文覆盖、包体和授权。
- 大标题可以紧凑、强对比和轻微错位，但正文不可模仿海报排版。长文必须稳定、舒展、容易扫描。
- 功能字不能承担长段中文；全大写英文只用于短标签，必须增加字距。

## 构图与材料语法

- 主要结构来自编辑排版而非通用组件库：目录、台账、页签、问题单、档案袋、票据、校样和边注。
- 用 1–2 px 靛蓝规则线建立秩序；分区优先用线、色面、留白和字号，不要给每一块都套容器。
- 圆角默认 0–8 px。仅圆形状态、呼吸工具、头像或系统控件可以使用完全圆形。
- 阴影是“错位印刷”的硬阴影：常用 4–10 px、无模糊或极少模糊。不要使用漂浮式柔光阴影。
- 允许极轻的纸纹、网格、装订线或套印偏移，但不能降低正文对比度，也不能覆盖大段阅读区。
- 垂直字、旋转 1° 左右的纸片、超大页码或边缘色块都是签名手法；每屏最多选一到两种。
- 图片不是默认依赖。需要插图时，优先平面色块、剪影、版画式线条或编辑摄影；不要用无关库存人物填空。
- 不要堆叠日本文化符号来“证明浮世绘”。品牌重点是平面色彩与编辑构成，而不是扇子、浪花、神社等表层装饰。
- 图标使用统一线宽的简洁几何图形；emoji 不能作为主要功能图标。

## App 布局原则

- 移动端优先，从 320 px 宽开始设计；同时验证常见手机、横屏和至少一个平板尺寸。
- 默认单列、纵向滚动。多列只在内容关系明确且真实设备测试通过时出现。
- 尊重刘海、状态栏、底部手势区和系统字体缩放。固定控件不能遮住正文、错误信息或最后一个操作。
- 每屏只保留一个主要动作。次要动作应降低层级，但不能靠极小文字或低对比隐藏。
- 导航名称必须表达目的，如“问题索引”“HRT”“前沿追踪”“工具箱”；禁止 `Dashboard`、`Explore` 这类空泛栏目。
- 不要照搬网站的桌面 Header。底部导航、顶部标题栏、分段控件或系统返回应按任务深度选择。
- 长页面需要可见的当前位置和可靠的返回路径；不要用装饰页码代替真实进度。
- 问题流要把“大问题”拆成可完成的小组，并允许保存进度、回看和修改，不用一次性长表单压迫用户。
- 结果页先给结论框架和适用边界，再给解释、来源与保存；不要先展示密集免责声明或技术字段。

## 组件与状态

- 可点击目标最小 44 × 44 pt/px；触摸项之间保持足够间距。
- 主按钮使用靛蓝实底或朱红实底，可配芥末金硬阴影。按下时应像纸张被压回桌面：轻微位移并收回阴影。
- 列表优先使用台账行、编号和规则线。只有确实需要分组或独立拖动时才使用卡片。
- 表单必须有持久可见的标签、说明、错误位置和单位；placeholder 不能代替标签。
- “是 / 否 / 不确定”等重复选择必须使用固定宽度和固定列位，使所有问题的答案在纵向上严格对齐。
- 步骤导航必须同时用文字、位置和高亮表示当前步骤；不可只靠颜色，也不可出现“点了但不高亮”。
- 空状态解释为什么为空，并给一个下一步；错误信息使用普通语言说明发生了什么以及用户能做什么。
- 生成的保存图必须像独立设计稿：清晰标题、日期/版本、适用边界、关键信息层级、来源入口和 `mtfbook.com` 二维码；禁止直接截取应用 UI 拼成长图。

## 动效语言

- 动效表达“翻页、推开、展开、套印归位和纸张压下”，不表达漂浮、弹跳或游戏奖励。
- 微交互一般为 160–220 ms；抽屉、折叠与问题页展开为 320–430 ms；首屏分段揭示可到 520 ms。
- 标准缓出曲线使用 `cubic-bezier(0.16, 1, 0.3, 1)`。
- 同一动作只使用一种主运动方向。展开应保持触发点与内容之间的空间因果关系。
- 动效必须服务于状态变化、定位或反馈；不要为静态文本逐字飞入，不要无限循环装饰，不要使用强视差。
- 完整支持“减少动态效果”：取消路径描画、位移和长过渡，直接呈现最终状态，且不损失信息。

## 温和模式与隐私表达

- 用户可见名称固定为“温和模式”。它可以替换敏感标题、通知预览或表面措辞，但不能暗示已经隐藏浏览记录、系统最近任务或网络日志。
- 温和呈现与真实隐私能力必须分开描述。只有实际实现本地存储、应用锁、模糊最近任务等功能后，才可宣称对应保护。
- 医疗、身份与问卷数据默认本地处理；未经用户明确授权，不上传、不加入分析事件、不写入第三方日志。
- 分享或导出前先给完整预览，并让用户决定是否包含姓名、日期、二维码和敏感回答。
- 提供快速退出时，说明它会做什么和不会做什么；不要用虚假的“一键无痕”承诺。

## 内容语气

- 使用直接、温和、具体的中文。先承认用户此刻的问题，再给可完成的下一步。
- 避免命令式诊断、身份审判、夸张保证、营销口号和空洞安慰。
- 不要把“年轻化”误写成俏皮话、网络梗或大量 emoji。尊重感来自清楚和诚实。
- 对不确定内容明确写出“目前不知道什么”“证据到哪里”“何时更新”。
- 按钮写动作和结果，如“查看监测说明”“保存方案图”，不要只写“确定”“继续”或“了解更多”。

## 医疗内容与 HRT 工具边界

- 每一项医疗结论都要有可访问的权威来源、来源机构、发布日期或版本、查阅日期和适用人群。
- 明确区分指南建议、研究发现、编辑解释和产品规则；不同证据不能混写成同一确定性。
- 未经单独评审的产品决策，禁止新增具体剂量、购买渠道、注射教学、自行调药方法或个体化化验解读。
- HRT 方案匹配必须是版本化、可审计、确定性的规则系统；同一输入必须得到同一结果。不可让生成式模型自由决定医疗方案。
- 结果必须解释每个组成项“是什么、为什么出现、适用边界是什么、需要与医生讨论什么”，并链接到原始指南。
- 不得把身高体重单独包装成足以决定 HRT 方案的依据。输入只收集规则确实需要且有来源支持的信息。
- 青少年、妊娠可能、血栓风险、急性危险信号和无法判断的回答必须有明确分支、停止条件或转介提示。
- 保存图不得把教育性匹配伪装成处方、医嘱或诊断证明。

## 可访问性

- 保持正文和交互控件达到 WCAG AA 对比度；任何状态都不能只靠颜色表达。
- 支持系统动态字体、屏幕阅读器、键盘/外接键盘、逻辑焦点顺序和清晰的可见焦点。
- Web 原型的焦点基准为 3 px 芥末金外框并留出间距；原生端使用等价且高对比的系统焦点/选中反馈。
- 标题层级、表单标签、分组说明和错误关联必须有可访问语义。
- 动态更新要向辅助技术说明结果，但不要让每次轻微输入都触发冗长播报。

## 工程与变更纪律

- 当前工程由 XcodeGen 2.46.0 的 `project.yml` 生成；修改项目结构后必须重新生成并提交 `.xcodeproj`。
- App 不含第三方运行时依赖。新增依赖前先更新技术决策，说明用途、替代方案、维护状态和隐私影响。
- 实现后端功能前，必须先调研可复用或可改造的开源项目与成熟解决方案，并评估其许可证、维护状态、安全性、隐私影响和与目标需求的匹配程度；优先采用或局部改造满足要求的方案，只有候选方案无法达成核心目标或会引入不可接受的风险与复杂度时才自行实现，并记录放弃候选方案的理由。
- 颜色、字号、间距、圆角、阴影、动效和断点必须集中为设计令牌；组件只能消费语义令牌。
- 业务规则与界面分离。医疗匹配规则应可在没有 UI 的情况下测试、版本化和审计。
- 不因一次页面需求引入通用框架；依赖必须说明用途、维护状态、隐私影响和替代方案。
- 不修改同级网站仓库来“配合” App，除非用户明确把网站也纳入当前任务。
- 不提交密钥、`.env*`、构建产物、依赖目录、生成报告、用户问卷样本或真实医疗数据。

### 唯一有效的工程命令

- 安装工程生成器：`brew install xcodegen`
- 生成工程：`xcodegen generate --spec project.yml`
- 打开并运行：`open Unmanual.xcodeproj`，选择 `Unmanual` scheme 和本地模拟器后运行；没有 Apple Developer 账号时只使用模拟器。
- 无签名编译：`xcodebuild -project Unmanual.xcodeproj -scheme Unmanual -configuration Debug -sdk iphonesimulator -destination 'generic/platform=iOS Simulator' -derivedDataPath .build/DerivedData CODE_SIGNING_ALLOWED=NO build`
- 单元测试：先用 `xcrun simctl list devices available` 取得测试设备 UDID，再运行 `xcodebuild -project Unmanual.xcodeproj -scheme Unmanual -configuration Debug -destination 'platform=iOS Simulator,id=<SIMULATOR_UDID>' -derivedDataPath .build/DerivedData CODE_SIGNING_ALLOWED=NO test`
- 当前没有独立 lint 或自动格式化依赖；以 Swift 6 编译警告、Xcode 格式和测试为基线。不要擅自引入格式化工具。

## 验证要求

任何界面任务在交付前至少完成：

1. 运行技术栈规定的 lint、类型检查、单元测试和构建；若命令尚未确定，明确标注未验证项。
2. 在 320 × 568、390 × 844、430 × 932 和 768 × 1024 等代表尺寸做真实渲染检查。
3. 完整点击主流程，覆盖返回、修改答案、空值、错误、加载、保存、取消和重新进入。
4. 检查触摸目标、文字缩放、横向溢出、键盘遮挡、安全区、屏幕阅读器标签和减少动态效果。
5. 对 HRT 规则运行确定性测试、边界分支测试和来源版本测试；随后做手机与大屏的浏览器/设备冒烟测试。
6. 保存图在至少一台窄屏手机上预览，验证中文断行、二维码可扫、来源可读且没有敏感信息意外泄漏。

### 本机空间与模拟器清理

- 当前开发机可用空间有限。优先复用现有 Simulator Runtime，不为单次验证重复下载运行时。
- 新建模拟器设备时记录其 UDID，并明确区分本项目设备与用户已有的共享设备。
- 每轮项目冒烟测试完成后，删除本项目新建的模拟器设备以及核对过的项目专属测试产物。
- 删除前先列出精确目标；不得用宽泛命令删除全部模拟器、全部 DerivedData 或共享 Runtime。
- Simulator Runtime 可能被其他项目共用，除非用户再次明确授权，否则不卸载。

## 禁用的视觉捷径

- 不做通用 SaaS Dashboard、全屏指标盘或“左侧栏 + 一排圆角卡片”。
- 不使用紫蓝霓虹渐变、玻璃拟态、毛玻璃、发光描边或无语义的 3D 球体。
- 不把 Bento Grid 当默认答案；只有内容关系真的需要并置时才使用网格。
- 不滥用圆角药丸、巨大空心图标、渐变按钮、emoji、轮播图和自动播放。
- 不为“高级感”降低字号、对比度或点击面积。
- 不复制网站首页的区块顺序，也不把网页 Header、问题门或内容地图原样搬进 App。
- 不用装饰编号冒充信息结构；每个编号必须对应真实顺序、版本、分类或定位。

## 完成定义

一个功能只有在以下条件同时满足时才算完成：它解决明确用户任务；视觉上属于上述纸本编辑系统；触摸和辅助技术可用；隐私与医疗边界写清；关键规则有测试；手机和平板真实渲染通过；没有用历史 V1 或网站布局充当捷径。
